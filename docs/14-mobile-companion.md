## 1. Product

**TokenFuse** — a native iOS command center for TokenFuse. Not a
dashboard-on-a-phone: an *enforcement* organ, consistent with the project's
"enforcement, not observability" positioning.

Core loop: agents run through tokenfuse gateways → gateways stream telemetry to
the Cloud control plane → the phone shows live burn rate (Dynamic Island /
Live Activity), pushes anomalies (budget %, 402/403, loop, kill), and lets the
operator **kill a run or clamp a budget with Face ID**, with mutations signed
by a Secure Enclave key.

Distribution strategy (per [09-product-strategy.md](09-product-strategy.md)):
the app is **free** and sells TokenFuse Cloud. Pro IAP deferred until there is
someone to convert.

Marketing beats (feed [content-shorts pipeline]): Dynamic Island burn rate,
Face-ID kill, Action-Button "physical agent kill switch", Apple Watch kill.

└───────▲──────────────▲─────────────────────────────┘
        │ HTTPS + SSE  │ APNs (alerts + LiveActivity)
┌───────┴──────────────┴─────────────────────────────┐
│ crates/cloud — Rust control plane (axum)           │
│ ├─ ported Go parity: ingest/runs/summary/kill/     │

│ ├─ NEW: /v1/pair, /v1/devices, signed-mutation     │
│ │  middleware, /v1/stream (SSE), /v1/series,       │

        │ CloudSink batches CallRecord (existing)

Hosting requirement (new, operational): the control plane must be reachable
24/7 for APNs. Options: fly.io hobby tier or Hetzner; image already ships via
GHCR. Decision deferred to Phase C (dev/beta can run on any box or tunnel).

## 4. Protocol specifications

Implementers: follow these exactly; do not improvise crypto or canonicalize
differently.

### 4.1 Pairing

1. Admin (dashboard or CLI) calls `POST /v1/pair/new` (admin key) →
   `{ "code": "<8-char one-time>", "expires_unix": … }` (TTL 10 min, one use).
2. Dashboard renders QR: `{"url":"https://plane.example","code":"…"}`.
3. App scans QR, generates key in Secure Enclave (P-256), calls
   `POST /v1/pair` `{code, pubkey_b64 (X9.63/SEC1), platform:"ios",
   name:"Yurii's iPhone"}` → `{device_id, org, role, device_token}`.
4. `device_token` (random 256-bit, Bearer) authorizes **reads**; the Enclave
   key authorizes **mutations**. Store both in Keychain
   (`kSecAttrAccessibleWhenUnlockedThisDeviceOnly`).

### 4.2 Signed mutations (kill / budget / device management)

Headers on the request:

```
Authorization: Bearer <device_token>
X-Fuse-Device: <device_id>
X-Fuse-TS:     <unix seconds>
X-Fuse-Nonce:  <uuid v4>
X-Fuse-Sig:    <base64 ES256 signature>
```

String to sign (LF-joined, exactly):

```
{METHOD}\n{PATH}\n{sha256(body) hex, empty body → sha256("")}\n{X-Fuse-TS}\n{X-Fuse-Nonce}
```

Encoding (fixed in A8): the signature is base64 of the **raw 64-byte `r||s`**
(IEEE P1363) form — exactly CryptoKit's `ECDSASignature.rawRepresentation`. The
public key submitted at pairing is base64 of the **SEC1/X9.63** bytes —
CryptoKit's `publicKey.x963Representation`. `{PATH}` is the request path only
(no query string).

Server verification: device exists ∧ role=admin ∧ |now−ts| ≤ 120 s ∧ nonce
unseen (LRU per device, cap 4096) ∧ ES256 signature valid against the stored
pubkey. Failure → `403 signature_invalid` (new error type, same envelope as
taint/dlp 403s).

**Simulator note:** Secure Enclave is unavailable in the iOS Simulator. The
crypto layer must be a protocol (`SigningKey`) with two impls: `EnclaveKey`
(device) and software `P256.Signing.PrivateKey` stored in Keychain
(simulator/DEBUG). Same wire format; server can't and needn't tell them apart
(App Attest hardens this in v2).

### 4.3 Push

- `POST /v1/devices/{id}/apns` `{token}` — register/refresh the APNs device
  token (signed).
- Alert pipeline (server): on ingest/kill/budget events emit push when:
  budget ≥ alert threshold (reuse existing `TOKENFUSE_CLOUD_ALERT_PCT` logic),
  `decision` field indicates a block (verify exact values in
  `crates/gateway/src/sink.rs` / proxy code before implementing), kill
  confirmed. Dedup: at most one push per (run, reason) per 10 min.
- Live Activity: app starts Activity → obtains ActivityKit push token →
  `POST /v1/devices/{id}/activity` `{activity_token, run_id}` (signed); server
  sends ActivityKit updates (spend, budget, pct, state) every 30 s while the
  run is live and on terminal events; ends the Activity on kill/completion.
- Env: `TOKENFUSE_APNS_KEY_PATH` (.p8), `TOKENFUSE_APNS_KEY_ID`,
  `TOKENFUSE_APNS_TEAM_ID`, `TOKENFUSE_APNS_TOPIC`
  (`com.taipanbox.tokenfuse`), `TOKENFUSE_APNS_ENV=sandbox|production`.
  Unset ⇒ push disabled, everything else works (fail-open, house style).
- APNs must be behind a trait (`PushSender`) with a recording fake for tests;
  `a2` used only in the real impl.

### 4.4 New read endpoints

- `GET /v1/stream` (SSE, viewer): events `run_update`, `kill`, `budget`,
  `incident`, heartbeat (`ping`) every 25 s. Intended to replace polling.
  Shipped on the server; the current app still polls the reads, and adopting
  SSE on the client is a pending follow-up (not in the OpenAPI, by design).
- `GET /v1/series?run=…|org&window=1h|24h&step=60s` (viewer): burn-rate
  buckets `[{t, cost_microusd, calls, blocked}]` aggregated from the durable
  store. Feeds Swift Charts.

## 6. Testing strategy

- **Rust:** unit per module; e2e in CI via docker-compose (gateway + plane + mock upstream from `bench/`); signed-mutation and SSE tests are plain reqwest tests.
- **iOS:** XCTest for stores, signing, client (URLProtocol mocks); no UI-test suite in MVP (manual smoke script in repo: `mobile/ios/SMOKE.md`); push payloads via `xcrun simctl push`.
- **Device matrix:** simulator (iPhone 16/17 class) for daily work; physical device required for: Secure Enclave path, real APNs, Live Activity pushes, Face ID.
- **CI jobs after Phase B:** existing Rust/cluster/radar/cloud(dashboard)/python/js + new `ios`.

## 7. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Swift/SwiftUI new to the codebase | XcodeGen + generated client minimize boilerplate; SwiftUI is declarative (Flutter experience transfers); keep third-party deps at ~zero |
| Enclave/push untestable in simulator | protocolized `SigningKey`/`PushSender` + simulator fallbacks designed in from PR B3/A9, not bolted on |
| App Review rejection (server-dependent app) | C1 demo mode + demo account in review notes; app must render a meaningful zero-state without a plane |
| ActivityKit update budget/throttling | 30 s cadence, server-side; terminal events sent immediately |
| Runner images may lag Xcode 26 | B1 acceptance says "newest available"; build must also pass on latest stable SDK (availability-gate any 26-only API) |
| macOS CI minutes | repo is public → free macOS runners |
